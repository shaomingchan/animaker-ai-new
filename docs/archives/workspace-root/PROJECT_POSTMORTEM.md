## ⚠️ 核心教训（血的教训）

**"3小时写完代码，5天修Bug"**

这是本项目最深刻的教训。急于上线，跳过测试和审查，导致：
- 支付功能完全不可用（致命）
- 用户体验极差（严重影响转化）
- 反复修改浪费大量时间
- CEO改得太累，团队效率低下

**如果重来一次**：
- 宁可多花1天做测试，也不要花5天修Bug
- 宁可晚1天上线，也不要上线后天天救火
- 质量 > 速度，这不是口号，是血的教训

**成本对比**：
- 正确做法：3小时开发 + 1天测试 = 1.5天
- 实际情况：3小时开发 + 5天修Bug = 5.5天
- **浪费了4天时间！**

**更深层的问题：没有PRD就开始写代码**

对比两个项目：

| 项目 | 启动方式 | 开发效率 | Bug数量 | 结果 |
|------|---------|---------|---------|------|
| **Animaker.AI** | 没有PRD，直接写代码 | 3小时写完 | 12个致命/严重Bug | 5天修Bug，质量6.5/10 |
| **GoodTrans** | 2次启动会，深度讨论PRD | 计划充分 | 预防性发现问题 | 开发高效，质量预期9/10 |

**GoodTrans的正确做法**：
1. **启动会1**：讨论产品定位、目标用户、核心功能
2. **启动会2**：6部门毒舌评审PRD，挑刺找问题
3. **PRD v2.0**：CEO逐条回应，更新PRD
4. **开始开发**：所有人对齐，方向明确

**Animaker.AI的错误做法**：
1. ❌ 没有PRD文档
2. ❌ 没有启动会
3. ❌ 没有需求讨论
4. ❌ 直接开始写代码
5. ❌ 边写边改需求
6. ❌ 分辨率选项与工作流不匹配（需求不清）
7. ❌ 支付流程没有设计（想当然）

**结果**：
- Animaker.AI：3小时开发 + 5天修Bug = **5.5天，质量6.5/10**
- GoodTrans：2天启动会 + 高效开发 = **预计3天，质量9/10**

**教训**：
- **磨刀不误砍柴工**：2天启动会看似慢，实际更快
- **PRD是地图**：没有地图就出发，必然迷路
- **讨论是投资**：启动会的讨论是最划算的投资
- **预防 > 修复**：启动会发现的问题，开发时就不会犯

### Bug #1: 支付充值不生效
**发现时间**: 2026-03-08
**影响**: 🔴 致命 - 用户付款后积分不增加，无法使用服务

**问题描述**:
用户完成Creem支付后，webhook收到通知，但积分没有增加到账户。

**根本原因**:
1. Checkout API没有传递`metadata.referenceId`（用户ID）
2. Webhook无法识别是哪个用户支付的
3. 使用了错误的Creem URL格式（简单支付链接而非API）

**错误代码**:
```typescript
// ❌ 错误：使用简单支付链接，无法传递用户ID
const checkoutUrl = `https://creem.io/payment/${productId}`;
return NextResponse.redirect(checkoutUrl);
```

**正确方案**:
```typescript
// ✅ 正确：使用Creem API创建checkout，传递用户ID
const response = await fetch('https://api.creem.io/v1/checkouts', {
  method: 'POST',
  headers: { 'x-api-key': apiKey },
  body: JSON.stringify({
    product_id: productId,
    metadata: { referenceId: session.user.id }
  })
});
const data = await response.json();
return NextResponse.redirect(data.checkout_url);
```

**预防措施**:
- 支付流程必须端到端测试（注册→支付→积分到账→创建任务）
- Webhook必须有日志记录，方便排查
- 关键业务逻辑必须有单元测试

### Bug #2: 未登录用户点击支付显示错误
**发现时间**: 2026-03-09
**影响**: 🔴 致命 - 用户以为网站崩溃了

**问题描述**:
未登录用户点击"Get Credits"按钮，看到JSON错误`{"error":"failed to create checkout"}`，而不是跳转到登录页。

**根本原因**:
1. API返回JSON错误，但前端期望重定向
2. 登录页没有处理`redirect`参数
3. 登录后没有自动回到支付页面

**错误代码**:
```typescript
// ❌ 错误：登录页硬编码callbackUrl
await signIn("google", { callbackUrl: "/dashboard" });
```

**正确方案**:
```typescript
// ✅ 正确：读取redirect参数
const searchParams = useSearchParams();
const redirect = searchParams.get('redirect') || '/dashboard';
await signIn("google", { callbackUrl: redirect });
```

**预防措施**:
- 所有用户路径必须测试（已登录/未登录）
- 错误信息必须用户友好，不能直接显示JSON
- 登录流程必须支持回跳

### Bug #7: 文件上传无大小限制
**发现时间**: 2026-03-09（代码审查发现）
**影响**: 🟡 严重 - 可能内存溢出

**问题描述**:
用户可以上传任意大小的文件，可能导致服务器内存溢出。

**根本原因**:
没有考虑边界情况。

**错误代码**:
```typescript
// ❌ 错误：直接下载，不检查大小
const photoBuffer = Buffer.from(await photoRes.arrayBuffer());
```

**正确方案**:
```typescript
// ✅ 正确：检查文件大小
const MAX_FILE_SIZE = 200 * 1024 * 1024; // 200MB
const photoSize = parseInt(photoRes.headers.get('content-length') || '0');
if (photoSize > MAX_FILE_SIZE) {
  return NextResponse.json({ error: 'File too large' }, { status: 413 });
}
```

**预防措施**:
- 所有用户输入必须验证
- 设置合理的限制
- 边界情况测试

## 第四部分：经验教训

### 教训 #1: 支付流程必须端到端测试
**场景**: 支付充值不生效
**教训**: 支付是核心功能，必须完整测试：注册→支付→积分到账→创建任务

### 教训 #2: 不要跳过三审制度
**场景**: COO自己写代码直接部署
**教训**: 三审制度是质量保证，不能因为"是自己写的"就跳过
**行动**: 代码交付后必须走完整流程：Dev自审→多模型审查→部署

### 教训 #3: 需求变更必须更新PRD
**场景**: 分辨率选项与工作流不匹配
**教训**: PRD是唯一标准，变更必须同步
**行动**: 建立PRD更新流程，变更必须通知所有人

### 教训 #4: 用户路径必须全部测试
**场景**: 未登录用户点击支付显示错误
**教训**: 不同用户状态有不同路径，都要测试
**行动**: 建立用户路径清单：已登录/未登录/有积分/无积分
